# Security and Privacy

## Security posture for MVP

This is a hosted MVP with synthetic data only. No real patient data exists in the system. The platform demonstrates production-minded security thinking without requiring full HIPAA compliance.

## Data classification

- **All patient data is synthetic** — generated by deterministic seed pipeline
- **No real PHI** exists in the database
- **A banner or disclaimer** should clarify this to users ("Demo mode" badge)

## Auth model for hosted MVP

- Mock auth with predefined demo users (clinician, educator, admin, patient, caregiver)
- Cookie-based session (`t1d_demo_user`) — no login/password flow
- User switching via header dropdown
- **No auth gate** — publicly accessible since all data is synthetic
- Role checks exist in code (`requireRole`, `hasRole`) for architectural correctness
- Future: replace with NextAuth.js or Clerk when real auth is needed

## Required foundations

- role-aware access checks in code
- auditability for important reads/writes
- structured error handling in all routes
- environment-safe configuration handling
- no secrets committed in version control

## Privacy posture

- consent/privacy markers surfaced (ConsentRecord badges on patient detail)
- traceability for sensitive actions (AuditEvent model)
- no unsupported clinical output
- data provenance preserved for generated content

## Hosted environment considerations

- DATABASE_URL must use SSL in production
- No real credentials in `.env.example`
- Health check at `/api/health` verifies DB connectivity
- Synthetic data disclaimer visible in UI

## Design expectations

- architecture supports real auth replacement later
- privacy-aware decisions documented in decision log
- consent model is extensible for real consent flows
